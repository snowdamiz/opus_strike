# Current implementation context
From gameStore.ts lines 244-286 (updateGameState function):
```typescript
updateGameState: (state) => {
  const { playerId } = get();
  const players = new Map<string, Player>();  // <-- NEW MAP EVERY TICK

  for (const snapshot of state.players) {
    const existingPlayer = get().players.get(snapshot.id);
    if (existingPlayer) {
      players.set(snapshot.id, {
        ...existingPlayer,
        position: snapshot.position,
        velocity: snapshot.velocity,
        lookYaw: snapshot.lookYaw,
        lookPitch: snapshot.lookPitch,
        health: snapshot.health,
        state: snapshot.state,
        movement: snapshot.movement,
        abilities: snapshot.abilities,
        hasFlag: snapshot.hasFlag,
      });
    }
  }

  const localPlayer = playerId ? players.get(playerId) ?? null : null;

  set({
    tick: state.tick,
    serverTime: state.serverTime,
    gamePhase: state.phase,
    redScore: state.redScore,
    blueScore: state.blueScore,
    redFlag: state.redFlag,
    blueFlag: state.blueFlag,
    roundTimeRemaining: state.roundTimeRemaining,
    players,  // <-- NEW MAP REFERENCE TRIGGERS SUBSCRIBERS
    localPlayer,
  });

  // Update visual store with authoritative server positions for interpolation
  players.forEach((player, id) => {
    setPlayerVisualPosition(id, player.position);
    setPlayerVisualRotation(id, player.lookYaw);
  });
},
```

The problem: Creating a new Map (`const players = new Map()`) on every server tick changes the Map reference, which triggers all subscribers to re-render even if only positions changed.

# What needs to change
Two approaches possible:

**Option A (preferred):** Update existing Map entries in-place
- Keep the same players Map reference
- Update existing player objects in-place for position/rotation
- Only create new Map when players are added/removed
- This preserves the Map reference and prevents re-renders on position-only updates

**Option B (alternative):** Fix OtherPlayers subscription
- Keep creating new Maps (current behavior)
- Change OtherPlayers to use a selector that doesn't re-render on Map reference changes
- More complex selector logic, but preserves current updateGameState structure

We'll implement Option A as it aligns with the visualStore architecture: position data should be high-frequency (visualStore) and not trigger React re-renders. The gameStore should only trigger re-renders on structural changes (add/remove players) or game events (health changes, flag capture, etc.).

Reference: 02-02-SUMMARY.md established that OtherPlayers reads from visualStore for interpolation, so gameStore position updates are only needed for the authoritative snapshot - not for triggering re-renders.
